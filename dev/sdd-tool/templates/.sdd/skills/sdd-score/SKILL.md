# 스펙 품질 자동 스코어링

## 언제 실행하는가

- `/sdd:spec-score <feature>` — 수동 호출
- spec-validate 이후 정량 평가가 필요할 때
- 회고(retro) 데이터 반영하여 가중치 캘리브레이션 시

---

## 점수 체계

### 종합 점수 = 내부 품질(40%) + 스티어링 정합성(60%)

```
종합 = (내부 × 0.4) + (스티어링 × 0.6)
```

### 내부 품질 (40%) — 스펙 자체의 완성도

| 항목 | 비중 | 평가 기준 |
|------|------|-----------|
| 충돌 검사 | 15% | 요구사항 간 모순, design↔tasks 불일치, 중복 REQ |
| 모호성 검사 | 15% | "빠르게", "쉽게", "적절히" 등 측정 불가 표현; AC 없는 REQ |
| 명백한 오류 | 10% | 존재하지 않는 REQ 참조, 순환 의존성, 타입 불일치 |

### 스티어링 정합성 (60%) — 프로젝트 방향과의 일치도

| 항목 | 비중 | 평가 기준 |
|------|------|-----------|
| Product 정합성 | 30% | 목적/범위/핵심 가치와 정렬, 스코프 초과 여부 |
| Tech 정합성 | 20% | 기술 스택/패턴/제약과 일치, 금지 기술 사용 여부 |
| Structure 정합성 | 10% | 디렉토리/네이밍/모듈 구조와 일치 |

---

## 실행 절차

### 1. 가중치 로드

```
.sdd/settings/score-weights.json 읽기
→ 없으면 기본값 사용
→ 회고 캘리브레이션 반영된 값이면 그대로 사용
```

### 2. 스펙 파일 로드

```
.sdd/specs/<feature>/requirements.md
.sdd/specs/<feature>/design.md
.sdd/specs/<feature>/tasks.md
```

하나라도 없으면 해당 항목 0점 + 경고.

### 3. 스티어링 파일 로드

```
.sdd/steering/product/references/product.md (또는 SKILL.md)
.sdd/steering/tech/references/tech.md
.sdd/steering/structure/references/structure.md
```

없으면 스티어링 정합성 = N/A (내부 품질만으로 평가).

### 4. 내부 품질 평가

#### 충돌 검사 (15%)
- [ ] 요구사항 간 모순 탐지 (같은 기능에 상반된 동작 정의)
- [ ] requirements.md REQ ID ↔ design.md 참조 일치
- [ ] design.md 컴포넌트 ↔ tasks.md 태스크 매핑 완전성
- [ ] 중복 REQ 탐지 (의미적 유사도)

**채점:** 문제 0개 = 100%, 1개 = 70%, 2개 = 40%, 3개+ = 10%

#### 모호성 검사 (15%)
- [ ] 측정 불가 표현 탐지: "빠르게", "쉽게", "적절히", "충분히", "가능한 한"
- [ ] AC 없는 REQ 탐지
- [ ] AC가 테스트 가능한지 (구체적 수치/조건 포함 여부)
- [ ] 미정의 약어/용어 사용

**채점:** 문제 0개 = 100%, REQ당 1개 이하 = 70%, 2개 이하 = 40%, 초과 = 10%

#### 명백한 오류 (10%)
- [ ] 존재하지 않는 REQ ID 참조
- [ ] 순환 의존성 (A→B→A)
- [ ] design.md에 정의 안 된 컴포넌트가 tasks.md에 등장
- [ ] 파일 간 버전/날짜 불일치

**채점:** 문제 0개 = 100%, 1개 = 50%, 2개+ = 0%

### 5. 스티어링 정합성 평가

#### Product 정합성 (30%)
- [ ] 모든 REQ가 product 목적/범위 내에 있는가
- [ ] 핵심 가치(사용자 경험, 성능 등)를 REQ가 반영하는가
- [ ] 스코프 바깥 기능이 포함됐는가 (feature creep)
- [ ] 우선순위와 REQ 순서가 일치하는가

**채점:** 완전 정렬 = 100%, 1개 벗어남 = 70%, 2개 = 40%, 3개+ = 10%

#### Tech 정합성 (20%)
- [ ] 지정 기술 스택 사용 여부 (예: TypeScript, Express, SQLite)
- [ ] 금지 기술 미사용 여부
- [ ] 아키텍처 패턴 준수 (예: 모듈러 모놀리스)
- [ ] 외부 의존성 정책 준수

**채점:** 완전 일치 = 100%, 1개 위반 = 50%, 2개+ = 0%

#### Structure 정합성 (10%)
- [ ] 디렉토리 구조 준수
- [ ] 네이밍 컨벤션 준수
- [ ] 모듈 분리 기준 준수

**채점:** 완전 일치 = 100%, 1개 위반 = 60%, 2개+ = 20%

### 6. 종합 점수 산출

```
내부 = (충돌 × w_conflict + 모호성 × w_ambiguity + 오류 × w_error) / (w_conflict + w_ambiguity + w_error)
스티어링 = (product × w_product + tech × w_tech + structure × w_structure) / (w_product + w_tech + w_structure)
종합 = 내부 × w_internal + 스티어링 × w_steering
```

### 7. 결과 출력

```markdown
# 스펙 품질 스코어: <feature>

## 종합: XX/100 (등급)

| 카테고리 | 점수 | 비중 | 가중 점수 |
|----------|------|------|-----------|
| 내부 품질 | XX | 40% | XX |
| 스티어링 정합성 | XX | 60% | XX |

### 내부 품질 상세
| 항목 | 점수 | 문제 |
|------|------|------|
| 충돌 검사 | XX | (문제 목록) |
| 모호성 검사 | XX | (문제 목록) |
| 명백한 오류 | XX | (문제 목록) |

### 스티어링 정합성 상세
| 항목 | 점수 | 문제 |
|------|------|------|
| Product | XX | (문제 목록) |
| Tech | XX | (문제 목록) |
| Structure | XX | (문제 목록) |

## 등급
- A (90~100): 즉시 구현 가능
- B (70~89): 경미한 수정 후 구현
- C (50~69): 상당한 수정 필요
- D (0~49): 재작성 권장

## 개선 제안
1. (가장 큰 감점 요인부터 나열)
```

### 8. 결과 저장

```
.sdd/specs/<feature>/score.md — 스코어 리포트
```

---

## 사후 캘리브레이션 (회고 연동)

회고 데이터(`.sdd/retrospectives/`)에서 실제 문제 발생 패턴 분석:
- 충돌로 인한 실제 버그 → 충돌 가중치 ↑
- 모호성으로 인한 재작업 → 모호성 가중치 ↑
- 스티어링 위반으로 인한 방향 수정 → 해당 스티어링 가중치 ↑

캘리브레이션 결과 → `score-weights.json` 업데이트.

```
/sdd:spec-score --calibrate
→ 전체 회고 데이터 분석 → 가중치 자동 조정
```
